# 0) 当前产品主线

当前主线已经从“平面白盒样板”切到：

**`PCVR 专属宿主 + 无限房间基地游戏 + 本地叙事 runtime`**

第一阶段不做开放式无限内容，而是先做一个可扩展的 `主神大厅 / 主神空间` 基地壳。Unity 是唯一宿主，技术基线是：

- `Unity 6 LTS`
- `OpenXR`
- `XR Interaction Toolkit`
- `Input System`
- `PCVR only`

当前 1.0 目标是证明：

- XR 宿主能跑通基础输入、交互、移动和基地壳
- Runtime / Narrative / SQLite / Audit 继续存在，不因 XR 改造被推倒
- 玩家第一次进入出生在大厅中央神光球附近
- 商店壳、扩展门位、Avatar 区域和后续世界扩展接口成立

# 1) Fate Engine 的产品原则（写在墙上那种）

## P1. 默认可玩

拖进场景的东西必须“立刻能用”，哪怕是灰盒占位资源。

## P2. 自动化优先，手工是例外

凡是高频通用劳动：导入、绑定、LOD、散布、巡逻、掉落、技能拼装、Boss循环——都应有自动流程。

## P3. 白盒可审计

AI/程序化生成结果必须可解释、可 diff、可回滚、可复现（seed + recipe + lockfile）。

## P4. 规则与校验兜底

AI 负责提案与装配；**校验器负责真理**（性能预算、可达性、穿插、缺依赖）。

## P5. 20% 创作不受限

永远给开发者留出“独特逻辑”的扩展钩子，而且调试体验要好。

# 2) “80% 自动化”到底覆盖哪些内容？（划定边界）

## 80%（引擎应该默认提供的工业件）

### A. XR 基地与内容铺设

* 主神大厅 / 无限房间基地壳
* 中央神光球交互
* 商店壳 / 扩展门位 / Avatar 区域
* 基地状态展示、任务展示、命运状态展示

以及后续可扩展内容：

* 生物群落/森林/草地/岩石/道路边缘自动散布（recipe + seed）
* 自动 LOD/HLOD/impostor、预算约束
* 氛围套件：灯光、环境音、雾、粒子、天气（基础版）
* 第一阶段先不追求自研渲染，所有表现层都交给 Unity

### B. 通用交互积木

* 门、梯子、按钮、开关、触发区、容器、拾取、水域、传送、检查点
* 统一契约：Interact/State/Event + Saveable
* 角色基础组优先：walk / run / jump / ladder / pickup / throw

### C. NPC 基础行为、叙事与规则系统

* 巡逻/追击/返回/调查（模板 + 参数）
* 掉落系统：权重、伪随机、保底、等级缩放（数据驱动）
* Boss 模板：阶段(HFSM) + 技能池(Utility) + 节奏规则
* Runtime 对话、命运线、任务推进、快照回放、SQLite + audit

### D. 技能与动画复用（人形生态）

* Humanoid 语义骨架 + 自动重定向 + IK 修正
* “技能包（Ability Set）”装配：跳跃/攀爬/翻滚/打球等
* 角色基础组先以现实单位为标准进行参数化

### E. 工程化保障

* 包管理/Registry/lockfile（可复现）
* 校验器（可达性、穿插、缺参数、预算）
* Profiler/Trace/回放（可调试）

## 20%（开发者该专注的独特价值）

* Boss 独特机制：弱点、场地互动、脚本桥段、剧情演出
* 主角手感：连招、判定、反馈、镜头语言
* 关卡设计：节奏、叙事、谜题、奖励布局
* 美术风格与角色表演（尤其近景）
* 具体战斗系统的独特规则

# 3) 你要提供的“工具”是什么？（20% 能做好靠工具）

这部分是 Fate Engine 的第二护城河：让开发者做 20% 时不痛。

## 必备工具清单（按价值排序）

1. **可视化调试**：当前状态、事件链、技能选择原因（Utility 分数/权重）
2. **时间线回放**：最近 N 秒发生了什么（为什么触发某技能/掉落）
3. **可视化参数调优**：热更新参数、立即看到效果
4. **校验报告**：错误定位到对象与规则（可一键修复部分问题）
5. **组合模板编辑器**：把“篮球场/仓库区”这种复合积木做成资产

# 4) AI 在这套 80/20 架构里的“正确职责”

现在产品边界已经升级为“两套 AI 系统并存”：

* 编辑器 AI：负责 recipe、装配计划、参数建议、白盒导出
* 运行时 NPC AI：负责游戏内 NPC 的目标选择、对话和短中期行为规划

两者必须严格隔离，不能混成同一套系统。

* 输入：自然语言意图 + 约束（平台预算、风格、密度、玩法留白）
* 输出：**Recipe（配方）/Assembly Plan（装配计划）**

  * 用哪些积木包（版本+依赖）
  * 怎么接线（事件图）
  * 参数建议（含理由）
  * seed（可复现）
* 然后：跑校验器 → 出报告 → 人类微调

AI 不负责“最终正确”，它负责把你从 30 分钟拖拽接线降到 30 秒。

---

# 5) 让 80/20 成为“工程现实”的三件关键机制

没有它们，你会退化成“很多模板 + 一堆例子”的普通工具。

## M1. 插槽化资源绑定

所有积木/技能都用 Slots：动画、音效、特效、碰撞、提示 UI 都是插槽。

## M2. 版本化与可复现

任何生成/装配都输出：

* recipe + seed
* lockfile（积木包版本与哈希）
* 校验报告

## M3. 自动化测试场景

每个积木包必须自带 demo scene + 验证用例。
否则生态一大就会质量崩盘。

# 7) 当前 1.0 最小闭环（XR 版）

为了让产品定位真正站住，当前最值得做的是：

* `XR Base Hall`：主神大厅、神光球、商店壳、扩展门位、Avatar 区域
* `本地叙事 runtime`：规则 + 本地 LLM、SQLite、审计日志、快照/回放
* `Narrative Debug`：调试会话、候选、拒绝理由、命运变化、任务变化
* `真实 3D Avatar 基线`：扫描/拟合接口、有限微调、外显切换预留

做完这四项，Fate Engine 的 XR / PCVR 主线才算真正成形，而不是停留在平面白盒工具阶段。
