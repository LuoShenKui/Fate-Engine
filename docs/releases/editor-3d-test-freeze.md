# Editor 3D 测试冻结说明

## 冻结目标
为 Editor 3D 测试阶段建立统一冻结规则，确保测试输入稳定、结果可复现，并为测试启动提供单一准入信号。

## 冻结范围
冻结范围覆盖以下四类能力，冻结窗口内不得发生非 blocker 级别变更：

1. **协议（Protocol）**：协议结构、字段语义、版本兼容约束。
2. **渲染（Rendering）**：渲染输出路径、关键画面基线、渲染后端配置。
3. **交互（Interaction）**：编辑器内核心交互流程、事件触发链路、输入映射。
4. **校验（Validation）**：运行时/编辑器校验规则、门禁脚本判定逻辑、报错分级标准。

## 冻结窗口变更约束
- 冻结窗口内**仅允许修复 blocker**（阻塞测试执行或导致结果失真的严重缺陷）。
- 冻结窗口内**禁止引入新功能**、禁止扩大需求范围、禁止非必要重构。
- 所有 blocker 修复必须附带影响面说明，并复用既有检查流程验证。

## 固定检查命令集合
冻结期间按固定顺序执行以下命令，不得删减：

```bash
make check-unit
make check-integration
make check-replay
make check-visual
make check-perf-scenes
```

## 统一报告要求
- 每轮冻结检查结束后，必须产出一次统一报告。
- 统一报告至少包含：
  - 本轮执行时间与执行人。
  - 固定命令集合执行结果（成功/失败、关键日志摘要）。
  - blocker 修复清单及回归结论。
  - 是否满足测试启动条件。

## 测试启动标签机制
- 当且仅当固定命令集合全部通过且统一报告结论为可启动测试时，打测试标签（示例：`test-ready-m1`）。
- 测试标签是测试启动的**唯一依据**；无标签或标签过期视为不可启动。
