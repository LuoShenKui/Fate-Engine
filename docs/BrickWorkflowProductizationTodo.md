# Brick Workflow Productization Todo

> 目标：把当前“开发者流可用”的积木系统，推进到更像 Unity / UE 的“点一下导入即可使用”的产品化工作流。
>
> 边界：本清单关注包管理、导入体验、即插即用能力、组合积木和参数编辑体验；不重复维护 3D Demo 底座门禁细节。

## 产品目标

- 导入一个积木后，编辑器里能看见、能放进场景、能直接使用。
- 导入一个组合积木后，不需要手工重新接线才能完成最小玩法。
- 对开发者来说，积木导入体验应接近 `npm install` 的确定性，同时对编辑器使用者来说体验应接近 Unity / UE 的内容浏览器导入。
- 积木导入后，系统必须自动告诉用户：能不能用、缺什么、需要补什么。

## 目标体验（一句话）

“在编辑器里搜索一个积木，点导入，导入后直接拖进场景就能跑；如果缺依赖、缺资源、缺技能包，系统会自动提示并给出修复路径。”

## 术语收敛

- `Brick Package`：类似 npm 包，是可安装、可版本化、可校验的积木单元。
- `Brick Instance`：包安装后，被拖入当前场景中的一个实例。
- `Composite Brick`：组合积木，例如篮球场、小屋、仓库区。
- `Ability Set`：一组能力/输入映射/动作规则，例如“篮球技能包”。
- `Brick Registry`：积木仓库/目录，既可以是本地目录，也可以是未来的在线服务。

## P0：统一“积木包”和“编辑器导入”的概念

### 目标

避免当前“导入 Recipe”和“安装积木包”是两套心智模型，用户看不懂入口区别。

### 任务

- [x] 在编辑器 UI 中明确区分：
  - `导入场景/Recipe`
  - `安装积木包/Import Brick`
- [x] 给 `Import Brick` 提供单独入口，而不是只靠命令行脚本。
- [x] 导入成功后，把安装结果同步到编辑器可见的本地 Registry 视图。
- [x] 导入失败时，直接展示结构化错误，而不是只在终端里报错。

### 完成标准

- 用户能分清“导入场景”和“安装积木包”。
- 不看文档，也能在编辑器里找到积木导入入口。
- 安装后能在面板中立即看到新积木。

## P1：把“安装一个积木包”做成类似 npm 的确定性流程

### 目标

让积木安装像包管理器一样可复现、可升级、可回滚。

### 任务

- [x] 编辑器内支持从本地 artifact / lockfile / 本地 registry 安装积木。
- [x] 支持从本地 JSON / lockfile 文件选择导入积木。
- [x] 安装结果统一写入项目级 lockfile 和本地 registry 索引。
- [x] 已安装积木列表随项目配方一起导出与加载，不依赖单机本地缓存。
- [x] 支持查看已安装积木的：
  - `id`
  - `version`
  - `license`
  - `dependencies`
  - `compat`
- [x] 支持“升级/卸载/回滚”最小流程。
- [x] 支持安装前 dry-run，先告诉用户会装什么、会冲突什么。

### 完成标准

- 安装一个积木包后，版本和依赖是确定的。
- 换一台机器按同一 lockfile 可以复现同样结果。
- 升级和回滚有明确入口，不需要手改 JSON。

## P2：安装后“立即可用”

### 目标

导入一个门，就能直接放门；导入一个开关，就能直接放开关；不需要先写脚本或补一堆隐藏步骤。

### 任务

- [x] 导入后自动注册到 `Brick Palette` / 内容浏览器。
- [x] 拖入场景时自动带默认参数和 fallback 资源。
- [x] 若积木具备交互契约，拖入场景后默认就是可用态。
- [x] 若存在必要依赖，导入后自动提示是否补装依赖。
- [x] 给“刚导入的积木”提供一个最小 demo/preview 入口。
- [x] 刚导入的交互积木可一键生成最小试玩场景，不需要手工接线。

### 完成标准

- 门积木：导入后可直接拖入场景并开关。
- TriggerZone / Switch / Ladder：导入后可直接放置并产生最小交互。
- 不需要用户手动读 manifest 才能第一次用起来。

## P3：支持“组合积木”

### 目标

让“篮球场”“小屋”“仓库区”这种东西，不只是资产集合，而是带交互、氛围、技能和规则的复合包。

### 任务

- [x] 定义组合积木的安装后呈现方式：
  - 是否在 Palette 里单独显示
  - 是否可以展开查看内部子积木
- [x] 组合积木拖入场景时自动完成内部连线。
- [x] 支持组合积木的默认参数组和项目级覆写。
- [x] 支持组合积木安装后自动带 demo scene。
- [x] 组合积木导入失败时，能告诉用户缺的是哪个子包/资源/能力。

### 完成标准

- 导入一个篮球场，不是只导入模型，而是导入一套可玩的区域积木。
- 用户不需要手工把几十个子积木一个个拖进去接线。

## P4：支持技能包 / 能力授予

### 目标

验证“导入一个篮球场，再给人物搭配篮球技能，就可以投篮”这条路线。

### 任务

- [x] 定义 `Ability Set` 包格式和安装入口。
- [x] 支持角色/区域/组合积木给角色授予能力包。
- [x] 支持导入后检查：
  - 角色是否支持该能力包
  - 是否缺输入映射
  - 是否缺动作资源或动画槽位
- [x] 给“篮球场 + 篮球技能包”做最小闭环样例。

### 完成标准

- 导入篮球场包后，可自动提示是否需要篮球能力包。
- 安装篮球能力包后，角色进入区域可以获得“投篮”这类最小能力。
- 如果角色骨架/输入映射不兼容，系统能明确报错。

### 目标

### 任务

- [x] 增加 `Brick Registry` 面板：
  - 可搜索
  - 可筛选
  - 可查看版本
  - 可查看依赖和许可证
- [x] 增加 `Installed Bricks` 面板：
  - 已安装列表
  - 版本状态
  - 升级/卸载/回滚
- [x] 增加安装确认弹窗：
  - 将安装哪些包
  - 依赖链
  - 冲突项
  - 许可证
- [x] 增加安装结果面板：
  - 安装成功
  - 缺失项
  - 冲突项
  - 推荐修复动作
- [x] 重做编辑器整体布局，避免“按钮堆满工具栏”的调试感：
  - 左侧：资源/积木浏览与分类
  - 中间：3D 视口 / 场景
  - 右侧：属性、依赖、说明、校验结果
  - 底部：日志 / 输出 / 问题面板
- [x] 把高频操作从散按钮改成更符合工程软件习惯的入口：
  - 菜单栏
  - 顶部工具条
  - 右键菜单
  - Inspector
  - 面板标签页
- [x] 给积木补“介绍信息”展示，而不是只显示名字：
  - 用途
  - 版本
  - 作者/来源
  - 依赖
  - 默认能力
  - 是否可直接使用
- [x] 给导入/安装流程增加向导或分步弹窗，而不是只靠一个按钮触发全部流程。
- [x] 为错误、警告、缺失项设计更清楚的状态展示（而不是只在列表里输出一行文本）。

### 完成标准

- 常见安装/导入操作不依赖终端。
- 用户能在 UI 里看到“为什么这个积木现在不能用”。
- 编辑器在信息层次、布局和交互方式上更像 Unity / UE / DCC 工具，而不是调试页。

### 任务

  - engine compat

  - dependency
  - builtin resource
  - slot/defaults
  - ability/character compat
- [x] 错误面板展示“缺少哪些东西”。
- [x] 对常见问题提供修复按钮或建议动作：
  - 安装依赖
  - 补 fallback
  - 改参数
  - 切换兼容版本
- [x] 对组合积木提供“部分可用 / 完全可用 / 被阻断”三级状态。

### 完成标准

- 至少对高频问题能给出明确修复路径。

## P7：积木制作体验产品化

### 目标

让“制作一个积木”本身也足够简单，避免把复杂度从“做游戏”转移成“做积木配置”。

### 核心原则

- 做积木应该更像填表单和走向导，而不是手写一堆协议文件。
- 系统要主动告诉制作者“还缺什么”，而不是让人自己猜。
- 打包应是一键动作，不是手动串多个命令。

### 任务

- [x] 新建积木向导
  - 选择积木类型：Door / Switch / TriggerZone / Ladder / Enemy / Composite / Ability Set
  - 填基础信息：名称、ID、版本、描述、分类、许可证
  - 选择默认模板，而不是从空白 JSON 开始
- [x] 缺项检查面板
  - 实时显示当前积木缺少哪些内容
  - 分类显示：元数据、依赖、资源、参数、兼容性、脚本入口
  - 区分阻断项和警告项
- [x] 一键打包
  - 在编辑器内直接触发 precheck + package
  - 打包前自动运行校验
  - 打包后展示 artifact、checksum、版本信息
- [x] 模板化创建
  - 提供常见积木模板：
    - 单积木：Door / Switch / TriggerZone / Ladder
    - 组合积木：Cabin / Warehouse / BasketballCourt
    - 行为包：Ability Set / Enemy Patrol Template
  - 模板应自动带最小字段、默认参数和 fallback
- [x] 错误提示与自动补全
  - 缺字段时提示应该填什么
  - 缺 slot 时提示建议资源类型
  - 缺依赖时提示可安装的候选项
  - 能自动补默认值的地方尽量自动补

### 完成标准

- 新建一个 Door 积木不需要手写 manifest.json。
- 新建一个积木时，系统能清楚告诉制作者还缺什么。
- 制作者能在编辑器里完成“填写 -> 检查 -> 打包”闭环。
- 常见积木能从模板快速起步，而不是每次从零配置。

## P8：按真实游戏开发流程补齐缺失模块

### 目标

让积木系统不只服务工程原型，也逐步贴近真实游戏开发流程，覆盖资产、行为、角色、区域、组合内容和发布使用链路。

### 当前最缺的模块

- [x] 资产导入工作流（第一版：Inspector 内支持从本地文件导入并绑定到 slot，项目内资源库可复用已导入资源）
  - 模型、动画、音效、特效导入后，如何快速绑定成积木
  - 导入后自动识别常见资源类型并填入 slot 候选
- [x] 角色与能力系统
  - 角色是否支持某个能力包
  - 输入映射、动作资源、骨架兼容性检查
  - 例如“篮球场 + 篮球技能包”的最小闭环
- [x] 敌人与行为模板
  - 巡逻、追击、返回、调查、基础攻击
  - 掉落规则、基础属性、仇恨范围
  - 让“导入一个敌人模板后可直接复用”成立
- [x] 组合积木与区域模板
  - 小屋、仓库区、篮球场、营地
  - 不只是模型组合，还包括默认交互、氛围、规则和能力授予
- [x] 更完整的参数和实例编辑
  - 区分包默认值、场景覆写、实例覆写（第一版已支持三层值展示、对应重置动作、实例覆写与项目保存）
  - 支持更清晰的参数分组和高级参数折叠（第一版已支持参数分组、折叠与覆写标识）
- [x] 更完整的校验层
  - 角色/能力兼容
  - 组合积木内部缺项
  - 导入后直接可用性状态
  - 资源引用完整性
- [x] 更像生产工具的内容浏览器（第一版）
  - [x] 缩略图/预览（高频积木与示例已接入真实截图来源，其余退回程序生成预览）
  - [x] 分类/标签/搜索
  - [x] 最近使用/推荐积木
  - [x] 示例场景入口
  - [x] 独立示例场景浏览面板
- [ ] Boss 模板系统（长期目标）
  - 基础 Boss 基座：血量、仇恨、阶段、技能调度、死亡/掉落
  - 技能包：冲刺斩、跳砸、召唤、远程投射、范围技
  - 阶段模板：P1 / P2 / P3、低血狂暴、召唤阶段、转场阶段
  - 独特机制扩展点：保留给具体 Boss 做差异化
  - 调试可视化：当前阶段、当前技能、技能选择原因、冷却状态
  - 目标：让“做一个复杂 Boss”从几个月的全定制开发，收敛成“基础 Boss + 技能包 + 阶段配置 + 少量独特机制”
- [ ] 玩家能力包系统（长期目标）
  - Ability Set：把“玩家会做什么”做成可安装、可授予、可回收的能力包
  - 示例：篮球技能包、武术技能包、攀爬技能包、潜行技能包
  - 区域/组合积木可触发能力授予，例如“进入篮球场后可投篮”
  - 角色兼容性检查：骨架、输入映射、动作资源、状态机扩展点
  - 目标：让玩家玩法不再完全写死在角色控制器里，而是可以模块化扩展
- [ ] 非强制接入与扩展边界（长期原则）
  - 积木系统和能力包系统必须是“可选增强”，不能变成唯一开发路径
  - 开发者必须能跳过默认模板，直接写自定义玩法逻辑
  - 默认模板应服务 80% 重复劳动，剩余 20% 必须保留自由扩展点
  - 允许项目只使用部分系统，而不是被迫全量采用
  - 目标：产品提供高复用工作流，但不绑死开发者的游戏架构

### 完成标准

- 美术/玩法/关卡设计不需要读协议文件才能把一个内容做成积木。
- 导入一个敌人模板后，能直接调整巡逻、属性和攻击方式。
- 导入一个组合积木后，不需要手工重接线才能开始试玩。
- 导入一个能力包后，系统能明确告诉用户角色是否兼容、缺少什么。
- 长期上，Boss 不再每次从零做起，而是能基于通用 Boss 模板做二次创作。
- 长期上，玩家能力可通过 Ability Set 模块化扩展，而不是全部硬编码在单一角色控制器中。
- 长期上，开发者可以“选择使用哪些系统”，而不是被引擎工作流强制绑定。
