# 发布者体验优先 ToDo（MVP）

> 目标不是先证明“引擎底层最强”，而是先证明：**普通开发者可以低痛苦地做出、调试、发布、复用一个积木**。

## 1. 目标与范围

本 ToDo 面向“普通开发者发布积木”的最小闭环，优先打通以下链路：

1. 创建积木
2. 本地预览/调试
3. 自动检查
4. 打包
5. 上传发布
6. 安装使用
7. 升级/回滚

该文档用于约束 Demo 范围，避免过早陷入“底层能力堆叠”。

## 2. 优先级路线图

### P0（必须先做）

- [ ] **脚手架**：5 分钟创建并跑通 `Hello Block`。
  - 一条命令生成模板（manifest + 示例逻辑 + 示例资源）。
- [ ] **本地预览器**：参数修改可即时反馈。
  - 至少包含日志区域：加载状态、参数解析、生命周期事件。
- [ ] **发布前自动检查**：本地拦截高频错误。
  - 资源缺失、参数类型错误、入口导出缺失、引擎版本不兼容。
- [ ] **一键打包**：统一产物格式（建议 `.fateblock`）。
  - 产物包含版本与完整性信息（如 hash）。
- [ ] **一键安装到另一个项目**：验证“可复用”而非“可上传”。

### P1（建议紧跟）

- [ ] **版本与兼容规则**：清晰 semver，安装时提示兼容风险。
- [ ] **发布流程**：`publish` 前自动执行校验，成功后返回可分享标识。
- [ ] **下载与更新**：支持升级、变更说明、快速回滚。

### P2（生态增长项）

- [ ] **模板中心**：UI、交互、数据驱动、特效等模板。
- [ ] **人话报错**：错误包含“问题 + 文件/位置 + 修复建议”。
- [ ] **样例优先**：提供可复制的最小样例集合。

## 3. “门积木”分层定义（避免抽象失焦）

“门积木”通常不是单一对象，而是一个小型制品包。建议固定三层抽象：

### 3.1 资源层（Resource Pack）

只包含可替换素材，不包含行为逻辑：

- 模型（门框/门板/把手）
- 材质与贴图
- 音效
- 动画片段

### 3.2 能力层（Behavior Pack）

定义通用行为，不绑定特定美术：

- `open()` / `close()`
- `lock()` / `unlock()`
- 事件回调（如 `onOpen` / `onClose`）
- 可挂接碰撞与交互接口

### 3.3 成品层（Product Pack）

资源 + 行为 + 默认配置组合，目标是“下载后可直接使用”：

- 默认模型/材质/动画/音效
- 默认碰撞盒、触发范围
- 默认交互面板与示例场景

## 4. 包元信息最低要求（发布准入）

每个积木包至少应声明：

- 包类型：`resource` / `behavior` / `product`
- 包内容清单：模型、材质、音效、动画、代码、配置
- 依赖清单：引擎版本、运行时能力、外部系统
- 暴露能力：可调用 API 与事件
- 适用场景：2D/3D、单机/联机、平台限制
- 许可信息：代码许可、素材许可、商用与再分发限制

## 5. Demo 验证清单（先做“窄门”）

为避免过度设计，先限定一个极小范围：

- 单机
- 3D
- 旋转门
- 一套模型与材质
- 两个音效（开/关）
- 一个开门动画
- 一个锁状态
- 一个统一交互接口

在该范围内验证：

- [ ] 资源是否可完整打包且不遗漏
- [ ] `open/close/lock` 接口是否可标准化
- [ ] 在另一个 Demo 项目中是否可安装并运行
- [ ] 缺依赖时是否给出可定位、可执行的错误提示
- [ ] 许可信息是否随包分发并可被消费端读取

## 6. 里程碑定义（发布者闭环）

当以下 9 步可稳定复现时，即视为当前阶段完成：

1. 创建积木
2. 配置元信息
3. 编写少量逻辑
4. 本地预览
5. 自动检查
6. 打包
7. 发布
8. 被另一个项目安装
9. 安装后可运行
